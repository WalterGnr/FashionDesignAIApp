# Worker And Queue Architecture

Last updated: 2026-07-09

Status: Sprint 08 planning artifact.

## Decision

Use Redis as the broker and a separate Python worker process. Start with Celery because render jobs need mature retries, acknowledgements, routing, worker concurrency controls, and operational visibility.

Keep the queue adapter behind an application interface so a future managed queue can replace Redis/Celery without changing API contracts.

## Flow

```text
Electron -> FastAPI POST /renders
  -> validate ownership and exact design version
  -> commit render job + immutable input
  -> enqueue job ID after commit
  -> return 202 immediately

Worker -> claim job lease
  -> build versioned prompt
  -> call provider adapter
  -> upload asset
  -> commit asset + succeeded status

Electron -> poll GET /renders/{id}
  -> later receive push notification when available
```

## Transactional Dispatch

Avoid the database/queue dual-write gap with an outbox table:

1. Commit render job and `render_job_requested` outbox event together.
2. Dispatcher publishes the event to Redis.
3. Dispatcher marks the outbox event delivered.
4. Worker processing remains idempotent by render job ID.

## Worker Rules

- one active lease per render job
- acknowledge queue work only after durable state transition
- cap concurrency separately from API concurrency
- set provider and storage timeouts
- do not log raw generated bytes
- check cancel state before provider call and before asset publication
- release/renew leases explicitly

## Initial Local Processes

- FastAPI service
- PostgreSQL
- Redis
- render worker
- optional outbox dispatcher, initially colocated with worker

Render work never runs inside the Electron renderer or FastAPI request thread.
