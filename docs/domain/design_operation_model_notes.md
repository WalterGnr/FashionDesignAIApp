# Design Operation Model Notes

Last updated: 2026-06-20

Status: Sprint 01 learning note. This is not implementation.

## Purpose

Design operations are the controlled internal language for changing a dress spec.

Natural language will come later through voice and AI. Sprint 01 should define the safe structured operations first.

## Core Principle

AI and UI should not mutate the dress spec directly.

They should request a design operation. The app validates the operation, applies it immutably, and creates a new version when appropriate.

## Conceptual Flow

```text
current DesignVersion
  + DesignOperation
  + validation rules
  -> new DesignVersion or ValidationError
```

## Operation Categories

### Set Field

Purpose:

- Replace a scalar or object field with a new value.

Examples:

- Set primary color to red.
- Set silhouette to mermaid.
- Set neckline to sweetheart.

Validation concerns:

- Field exists.
- Value matches allowed type.
- Field is not locked.
- Value is compatible with MVP scope.

### Add Detail

Purpose:

- Add an item to a list-like field.

Examples:

- Add pearl trim around neckline.
- Add thigh-high slit.
- Add lace applique to bodice.

Validation concerns:

- Target list exists.
- Detail has type and placement.
- Duplicate handling is defined.

### Remove Detail

Purpose:

- Remove a detail from a list-like field.

Examples:

- Remove bow from waist.
- Remove sequins from sleeves.

Validation concerns:

- Detail exists or graceful no-op behavior is defined.
- Locked fields are respected.

### Modify Measurement

Purpose:

- Set or update a numeric measurement with unit.

Examples:

- Set dress length to 58 inches.
- Add 2 cm to sleeve length.

Validation concerns:

- Unit is valid.
- Range is reasonable.
- Measurement target exists.
- Operation distinguishes absolute value from relative adjustment.

### Lock Field

Purpose:

- Protect a field from future edits.

Examples:

- Lock neckline.
- Lock fabric.
- Lock silhouette.

Validation concerns:

- Field path is valid.
- Lock reason may be recorded.

### Unlock Field

Purpose:

- Allow a previously locked field to change.

Validation concerns:

- Field is currently locked.
- Future user permission rules may apply.

### Create Variation

Purpose:

- Create a related branch from a version with one or more changes.

Examples:

- Show three sleeve options.
- Create a bridal version of this dress.

Validation concerns:

- Parent version is recorded.
- Variation label is recorded.
- Locked fields are preserved.

### Revert To Version

Purpose:

- Create a new version from an older version.

Important:

- Reverting should not delete history.
- It should create a new current version whose contents match the selected older version.

## Operation Metadata

Each future operation should probably record:

- Operation ID
- Operation type
- Actor: user, AI, system
- Source: voice, text, UI, import, system
- Timestamp
- Target field path
- Reason or raw command reference
- Confidence if generated by AI
- Whether it creates a new version

## Validation Result

A validation result should distinguish:

- Accepted
- Rejected
- Needs clarification
- No-op

Rejected examples:

- Unknown field
- Invalid value
- Locked field conflict
- Out-of-range measurement
- Unsupported garment category

Needs clarification examples:

- "Make it more elegant."
- "Use a better fabric."
- "Shorten it a bit."

## Version Creation Rule

Likely creates a new version:

- Set field
- Add detail
- Remove detail
- Modify measurement
- Create variation
- Revert to version

May create a new version or metadata-only event:

- Lock field
- Unlock field

Sprint 01 should decide this explicitly.

## Testing Implications

The future test suite should include:

- Operation applies expected change.
- Previous spec remains unchanged.
- Invalid operation is rejected.
- Locked field conflict is rejected.
- Revert creates a new version.
- Variation preserves parent version.
- Unknown and assumed values are preserved.
